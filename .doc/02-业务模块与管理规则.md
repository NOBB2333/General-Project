# 02 业务模块与管理规则

## 1. 修订说明

本文保留原编号，但按当前项目成熟后的理解重写，不再混合早期散装模块设想。

## 2. 业务模型

系统按三条稳定业务主线组织：

### 2.1 平台治理主线

解决：

- 用户
- 组织
- 角色
- 菜单
- 日志
- 监控
- 文件

### 2.2 项目执行主线

解决：

- 项目建档
- 项目详情
- 里程碑、周期、任务、问题、文档、成员
- PM 与 PMO 的执行视图

### 2.3 经营管理主线

解决：

- 经营总览
- 单项目经营下钻
- 经营报表和预警

## 3. 当前业务模块划分

### 3.1 平台中心

当前承接：

- 组织架构
- 用户管理
- 角色权限
- 菜单管理
- 租户管理
- 文件管理
- 审计日志
- 系统监控
- 定时任务
- 个人中心

说明：

- `我的代办` 不再作为平台中心正式菜单

### 3.2 项目执行区

当前承接：

- 项目列表
- 项目详情
- 与我相关
- PM 工作台
- PMO 执行总览

### 3.3 经营管理区

当前承接：

- 经营总览
- 项目经营
- 报表中心

## 4. 核心对象

### 4.1 平台对象

- 组织
- 用户

- 菜单
- 文件
- 审计日志
- 定时任务

### 4.2 项目执行对象

### 4.3 经营对象

## 5. 管理规则

### 5.1 菜单规则

- 菜单应按分区组织，不按散装对象平铺
- 一级菜单必须控制数量
- 平台菜单不承接业务驾驶舱页面

### 5.2 页面规则

- 项目执行区以 `项目详情` 为核心
- 经营管理区以 `经营总览 / 项目经营 / 报表中心` 为核心
- 报表中心负责跨项目检索，不负责替代项目详情和项目经营页

### 5.3 数据规则

- 指标口径统一由后端定义
- 预警类对象优先结构化，不在多个页面各自拼装
- 敏感金额和利润信息必须按权限控制

## 6. 当前结论

系统业务模块不是越多越好，而是必须围绕：

- 平台治理
- 项目执行
- 经营管理

三条主线稳定展开。
