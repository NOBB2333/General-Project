# 01 产品范围

## 1. 产品定义

产品名称暂定：`General Admin 企业内部项目管理平台`

这是一个面向企业内部人员的私有化项目管理平台，目标是把项目从立项、规划、执行到验收、开票、回款跟踪放到统一平台内管理。

产品第一优先面向研发交付型项目，而不是通用 OA 台账系统。

也就是说，这个平台首先要解决研发、测试、发布、验收和经营节点在一个系统里的协同问题，再逐步增强经营分析和智能能力。

当前产品形态建议先按“统一平台壳 + 逻辑分区”设计，而不是一开始就决定拆成多个独立系统。

建议的逻辑分区如下：

- 平台中心：承接登录、组织、权限、审批、通知、附件、日志等共享能力
- 项目执行区：承接项目立项、计划、需求、任务、测试、发布、风险、问题、变更和执行看板
- 经营管理区：承接预算、合同、验收、开票、回款和经营分析
- 智能分析区：承接 AI 摘要、风险提示、周报辅助生成和智能洞察

后续如果业务边界足够稳定，再把这些逻辑分区逐步强化为多个业务 APP。

核心使用角色：

- PMO
- 项目经理
- 项目成员
- 财务接口人
- 审批负责人
- 审计人员
- 管理层查看者

其中高频核心用户主要是：

- PMO：看多项目状态、异常项目、预算偏差、合同节点、回款预警
- PM：看自己项目的里程碑、需求、任务、测试发布节奏、阻点、风险问题和执行节奏

产品默认支持三类项目执行模式：

- 瀑布模式：按阶段、阶段门、交付物、评审和结项来治理
- 敏捷模式：按需求池、迭代、缺陷、版本和复盘来执行
- 混合模式：上层按阶段治理，下层按迭代执行

## 2. 解决的问题

这套系统主要解决以下问题：

- 项目推进过程分散在 Excel、聊天和口头沟通中，不易追踪
- PM 很难统一管理里程碑、任务、风险、问题和变更
- 研发项目中的需求、测试、发布和验收经常散落在多个工具里，缺少统一主线
- 晨会纪要、行动项、决策、依赖等治理信息往往留在聊天里，后续无法追踪
- PMO 很难看清多项目整体状态和异常项目
- 验收、开票、回款等关键节点缺少统一跟踪和预警
- 项目资料、合同资料和审计证据分散在附件、网盘和邮件中，归档与留痕不足

## 3. 产品边界

本产品要做的内容：

- 平台中心与共享能力
- 项目立项与项目模板
- 项目集、项目管理
- 项目阶段、里程碑、需求、任务、缺陷、待办
- 迭代、测试、发布、验收协同主线
- 风险、问题、变更管理
- 会议纪要、行动项、决策、依赖、假设管理
- 预算基础跟踪
- 合同执行基础跟踪
- 验收、开票、回款节点跟踪
- 审批、通知、评论、附件、日志留痕
- 项目资料和合同资料台账、归档与审计
- PM 看板和 PMO 看板
- 私有化部署和企业级交付

本产品当前不做的内容：

- 供应商协同平台
- 客户门户
- 采购系统
- 完整财务系统
- 移动端主产品
- 开放式 SaaS 协作平台

## 4. 主要使用场景

- 公司内部研发项目管理
- 内部交付项目管理
- 跨部门项目协同
- 重点项目跟踪和管理层查看
- 研发测试发布一体化跟踪
- 合同资料、验收资料和项目归档管理

不同角色关注点：

- PMO：看整体状态、异常项目、预算偏差、合同节点、回款预警
- PM：看自己项目的推进、阻点、风险问题、里程碑和执行节奏
- 成员：看自己负责的需求、任务、缺陷、待办和协同事项
- 管理层：看结果、看重点项目、看风险和异常

## 5. 产品主线

```text
统一平台壳
  ├── 平台中心
  ├── 项目执行区
  │     -> 项目立项
  │     -> 项目规划
  │     -> 需求与迭代
  │     -> 任务拆解
  │     -> 测试与发布
  │     -> 执行跟踪
  │     -> 风险/问题/变更
  ├── 经营管理区
  │     -> 预算跟踪
  │     -> 合同执行
  │     -> 验收/开票/回款
  │     -> 资料台账/归档
  └── 智能分析区
        -> 统计分析
        -> 风险提示
        -> 复盘总结
```

## 6. 范围控制

后续所有设计和研发都按以下原则控制范围：

- 一期优先保证可用，不追求做全
- 预算和合同在一期只做跟踪，不做财务核算
- 外部系统是主数据源时，本系统以获取和分析为主
- 重点项目用排序、标签、置顶和看板实现，不单独引入组合层级
- 一期优先打通研发交付闭环，不做完整 DevOps 平台

以下内容默认不进入一期：

- 财务核算
- 双向主数据写回
- 大而全报表中心
- 复杂 AI 能力
- 桌面端独立产品化
- 完整财务核算与采购协同
- 完整自由配置工作流平台

## 7. 当前结论

这个项目的本质不是做一个大而全的单体后台，而是做一个统一平台下、面向研发交付场景优先的企业内部项目管理产品，先按逻辑分区建设，再根据业务成熟度决定是否进一步演进成多个业务 APP。
