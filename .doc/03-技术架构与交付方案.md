# 03 技术架构与交付方案

## 1. 修订说明

本文按当前已经落地的工程实践修订，不再只停留在“建议选型”层面。

## 2. 总体架构

当前采用模块化单体架构，这一判断仍然成立。

原因：

- 项目仍处于业务边界逐步稳定阶段
- 私有化项目优先追求稳定和易维护
- 当前难点在业务结构和页面结构，不在物理拆分

## 3. 当前项目组成

### 3.1 后端

- `General.Admin`
- 技术栈：
  - `ABP`
  - `ASP.NET Core`
  - `EF Core`

当前开发环境默认数据库：

- `SQLite`

生产型方案仍可演进为：

- `PostgreSQL`
- `Redis`
- 对象存储

### 3.2 前端

- `General.Web`

当前主业务前端：

- `General.Web/apps/web-general`

平台中心形态：

- `General.Web/apps/web-general` 的 platform 运行/构建模式

前端技术基座：

- `vue-vben-admin`
- `Vue 3`
- `TypeScript`
- `Vite`
- `Ant Design Vue`

### 3.3 桌面端

- `General.Desktop`

当前只作为桌面壳预留，不作为主业务承接端。

## 4. 当前工程组织原则

### 4.1 后端

- 权限、菜单、平台治理能力优先沉淀在平台中心相关服务中
- 业务能力按分区逐步生长，不急于拆成多服务

### 4.2 前端

- 先在统一前端壳中承接平台、项目、经营三大分区
- 页面以业务控制台为目标，不做过多散装列表

## 5. 交付原则

### 5.1 Web 是当前正式主交付端

所有核心业务能力优先在 Web 中完整落地。

### 5.2 桌面端不是主产品线

桌面端只承担：

- 壳层承载
- 通知
- 文件选择增强
- 本地能力桥接

### 5.3 私有化优先

部署设计优先考虑：

- 单机可用
- 内网可用
- 易迁移
- 易备份

## 6. 当前技术结论

当前这套技术路线成立的关键原因不是“技术先进”，而是它能稳定支撑：

- 平台治理
- 项目执行
- 经营管理
- 后续 AI 能力预留
