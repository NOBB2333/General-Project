# 03 技术架构与交付方案

## 1. 总体架构

一期采用模块化单体架构。

原因：

- 当前目标是先做完整闭环
- 私有化项目更重视稳定、易部署、易维护
- 过早微服务会增加复杂度

在产品形态上，当前按三个项目组织：

- `General.Admin`：后端服务与业务能力
- `General.Web`：基于 `vue-vben-admin` 的 Web 管理端
- `General.Desktop`：基于 Avalonia 的桌面客户端壳，通过 WebView 承载 Web 端

在系统部署上，后端一期仍采用统一的模块化单体。

也就是说：

- Web 端先基于 `vue-vben-admin` 完成统一入口、权限菜单和分区切换
- 桌面端先做外壳，不重复建设业务页面
- 后端先不按物理微服务强拆
- 共享能力统一沉淀在平台中心
- 后续可根据业务成熟度把分区演进为更独立的 APP

## 2. 技术栈

后端建议：

- C#
- ASP.NET Core + ABP Framework(Layered Solution Template )
- EF Core
- PostgreSQL
- Redis
- SignalR
- Hangfire 或 Quartz
- OpenAPI / Swagger
- S3 兼容对象存储 / MinIO

abp new General.Admin \
  -t app \
  --layered \
  --database-provider ef \
  --ui none \          # 不生成任何 Web UI（我们用 Vben）
  --mobile none \      # 不生成移动端（强烈推荐）
  --public-website none \
  --multi-tenancy      # 启用租户支持
  
前端建议：

- `vue-vben-admin`
- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- ECharts

前端选型原则：

- 后台壳、布局、菜单、标签页、权限路由、表格表单基础能力优先复用 `vue-vben-admin`
- 非必要不自建后台基座，不再按原来的自搭后台方向推进
- 具体 UI 组件、主题体系和工程约定优先跟随 `vue-vben-admin` 当前版本能力

桌面端建议：

- Avalonia + WebView
- 项目名使用 `General.Desktop`

## 3. 系统模块划分

后端模块：

- 组织与用户
- 权限与认证
- 项目组合与项目集
- 项目中心
- 任务执行
- 风险 / 问题 / 变更
- 预算管理
- 合同执行
- 审批与通知
- 日志与审计
- 报表与统计分析
- 系统集成

前端逻辑分区建议：

- 平台中心
- 项目执行区
- 经营管理区
- 智能分析区

前端实现建议：

- 用 `vue-vben-admin` 承载统一登录后的主框架
- 逻辑分区通过路由分组、菜单分组和权限体系体现
- 高权限管理页、列表页、表单页、看板页优先沿用其工程规范

## 4. 前端与客户端交付

当前项目划分：

- `General.Admin`：ABP 后端主项目
- `General.Web`：Web 管理端主项目
- `General.Desktop`：桌面客户端壳项目

Web 是正式主交付端。

Web 前端建议采用基于 `vue-vben-admin` 的统一登录入口，再进入不同业务分区。

一期重点交付：

- 平台中心
- 项目执行区
- 经营管理区的轻量入口

二期重点增强：

- 经营管理区

三期重点增强：

- 智能分析区

桌面端只承担这些职责：

- 承载 Web 页面
- 系统通知
- 文件选择增强
- 自动更新
- 本地缓存

桌面端实现方式：

- 采用 Avalonia 做窗口壳与本地能力封装
- 业务主界面通过 WebView 承载 `General.Web`
- 桌面端与 Web 端之间只做有限桥接，例如下载、通知、文件选择和本地缓存

原则：

- 桌面端不是独立产品线
- 业务逻辑统一在后端和 Web 前端
- 各业务分区共享统一认证、权限、通知、附件、日志和审批能力
- Web 前端优先复用 `vue-vben-admin` 的菜单、权限、布局和工程能力

## 5. 系统集成

需要预留的对接边界：

- SSO、AD、LDAP
- ERP / 财务系统
- 企业微信、钉钉、飞书、邮件
- 对象存储
- GitLab、Jenkins、Azure DevOps

集成原则：

- 外部系统是主数据源时，本系统以获取和展示为主
- 一期不做复杂双向写回

## 6. 自动处理能力

系统应支持：

- 自动发送通知
- 自动催办审批
- 自动预算预警
- 自动回款预警
- 自动刷新统计数据
- 自动生成日报 / 周报基础数据

## 7. 数据存储与部署

数据存储：

- PostgreSQL：主业务数据
- Redis：缓存、会话、通知、限流
- 对象存储：附件、文档、导出文件

部署架构：

```text
Nginx / Gateway
  ├── Web 前端
  ├── ASP.NET Core API
  ├── SignalR 实时服务
  └── 后台任务服务

PostgreSQL
Redis
Object Storage
```

## 8. 私有化交付要求

- 支持单机部署
- 支持多节点部署
- 支持内网访问
- 支持离线环境部署
- 支持备份恢复
- 支持日志审计导出
- 支持基础资源监控
- 支持资源不足自动告警

## 9. 非功能要求

性能：

- 常规页面打开尽量控制在 3 秒内
- 列表查询尽量控制在 2 秒内
- 实时通知延迟尽量控制在 3 秒内

稳定性：

- 支持异常恢复
- 支持关键任务失败重试
- 支持后台任务补偿

安全性：

- HTTPS
- 权限校验
- 接口限流
- 防重复提交
- 敏感数据保护
- 文件权限控制

可观测性：

- 服务健康检查
- CPU、内存、磁盘监控
- 数据库、缓存、对象存储状态监控
- 异常告警

## 10. 当前架构结论

当前推荐路线：

```text
General.Desktop(Avalonia + WebView 壳)
            ↓
General.Web(vue-vben-admin 统一前端基座)
   ├── 平台中心
   ├── 项目执行区
   ├── 经营管理区
   └── 智能分析区
            ↓
General.Admin(ASP.NET Core + ABP Layered)
            ↓
PostgreSQL + Redis + Object Storage
```
