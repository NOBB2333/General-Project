# 会议纪要与治理闭环设计

## 1. 文档目的

本文件用于补齐项目治理过程中的结构化沉淀能力，把晨会、周会、专题会中的纪要、行动项、决策、依赖和假设纳入系统管理。

## 2. 设计原则

- 会议纪要不是普通评论，应形成可追踪的治理对象
- 纪要中的行动项应能转任务或待办
- 决策、依赖、假设要与风险、问题一起形成统一治理视图
- 一期以结构化沉淀和追踪为主，不做复杂会议门户

## 3. 核心对象

- 会议纪要
- 行动项
- 决策记录
- 依赖项
- 假设项

建议统一纳入 RAID 视图：

- Risk 风险
- Assumption 假设
- Issue 问题
- Dependency 依赖

## 4. 最小字段建议

### 4.1 会议纪要

- 纪要编号
- 会议类型
- 所属项目
- 所属阶段、里程碑或迭代
- 会议时间
- 主持人
- 参会人
- 纪要内容
- 记录人

### 4.2 行动项

- 标题
- 来源纪要
- 所属项目
- 责任人
- 计划完成日期
- 状态
- 关联任务或待办

### 4.3 决策记录

- 决策主题
- 所属项目
- 背景说明
- 决策结论
- 决策人
- 生效时间
- 影响范围

### 4.4 依赖项

- 依赖编号
- 所属项目
- 依赖类型
- 依赖对象
- 责任方
- 目标完成日期
- 状态
- 风险说明

### 4.5 假设项

- 假设编号
- 所属项目
- 假设内容
- 影响范围
- 验证日期
- 验证结果
- 状态

## 5. 核心流程

### 5.1 晨会纪要流程

```text
创建会议纪要
  -> 记录进展 / 阻点 / 决策
  -> 提取行动项
  -> 指定责任人与日期
  -> 后续跟踪关闭
```

规则：

- 晨会纪要可按日产生
- 行动项必须绑定责任人
- 阻点可转问题或风险

### 5.2 决策记录流程

```text
提出议题
  -> 会议讨论
  -> 形成结论
  -> 记录决策
  -> 通知相关人员
  -> 后续跟踪影响范围
```

规则：

- 关键决策应保留背景和结论
- 已生效决策默认不可物理删除
- 决策变更应形成新版本或补充记录

### 5.3 依赖与假设跟踪流程

```text
登记依赖或假设
  -> 指定责任方和目标时间
  -> 定期检查
  -> 若失效则升级为问题或风险
  -> 关闭
```

规则：

- 依赖应区分内部依赖和外部依赖
- 假设失效时必须有后续处理动作
- 超期依赖应进入 PMO 汇总视图

## 6. 页面建议

- 项目纪要列表
- 会议纪要详情
- 行动项列表
- 决策记录列表
- RAID 汇总页

项目详情页建议新增页签：

- 会议纪要
- 行动项
- 决策记录
- RAID

## 7. 权限建议

- 项目成员可查看所属项目纪要和行动项
- PM、PMO 可维护决策、依赖和假设
- 管理层默认查看只读摘要
- 敏感会议纪要应支持单独可见范围

## 8. 一期边界

一期做：

- 晨会、周会、专题会纪要
- 行动项转任务或待办
- 决策、依赖、假设基础记录
- RAID 汇总视图基础版

一期不做：

- 会议预约排程系统
- 音视频会议系统
- 自动语音转写
- 复杂纪要模板引擎

## 9. 本文件结论

项目治理闭环的关键不只是把任务做完，还要把会议、决策、依赖和假设留下结构化证据。这样 PMO 才能真正做跨项目治理，而不是只看结果。
