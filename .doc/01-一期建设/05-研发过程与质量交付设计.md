# 研发过程与质量交付设计

## 1. 文档目的

本文件用于补齐一期在研发交付场景下的过程设计，明确瀑布、敏捷和混合模式下，需求、测试、发布在系统中的最小落地方式。

## 2. 设计原则

- 一期优先建立需求到发布记录的主线闭环，不追求替代完整研发工具链
- 研发交付对象必须可挂接到项目、阶段、里程碑、任务和缺陷
- 瀑布、敏捷、混合三类模式共用同一套核心对象，不拆三套模型
- 测试和发布重点是留痕、可追溯、可汇总，不在一期做复杂流水线编排

## 3. 适用模式

### 3.1 瀑布模式

- 需求按阶段进入开发
- 测试计划跟随阶段或里程碑
- 发布记录按阶段验收节点归档

### 3.2 敏捷模式

- 需求进入需求池
- 需求按迭代排期
- 缺陷回流到迭代执行
- 发布记录按版本维护

### 3.3 混合模式

- 项目主计划按阶段治理
- 研发执行按迭代推进
- 发布结果同时回写项目阶段和迭代总结

## 4. 核心对象

一期建议新增以下对象：

- 需求
- 迭代
- 测试计划
- 测试报告
- 发布记录

对象关系建议：

```text
项目
  ├── 阶段 / 里程碑
  ├── 需求
  │     └── 任务 / 缺陷
  ├── 迭代
  │     ├── 需求
  │     ├── 任务
  │     └── 缺陷
  ├── 测试计划
  │     └── 测试报告
  └── 发布记录
```

## 5. 最小字段建议

### 5.1 需求

- 需求编号
- 标题
- 描述
- 来源
- 优先级
- 所属项目
- 所属阶段或迭代
- 验收标准
- 状态
- 负责人

### 5.2 迭代

- 迭代名称
- 所属项目
- 开始日期
- 结束日期
- 迭代目标
- 负责人
- 状态

### 5.3 测试计划

- 测试计划编号
- 所属项目
- 所属阶段、迭代或版本
- 测试范围
- 负责人
- 计划开始日期
- 计划结束日期
- 状态

### 5.4 测试报告

- 所属测试计划
- 报告编号
- 通过结论
- 缺陷统计
- 风险说明
- 报告时间
- 提交人

### 5.5 发布记录

- 发布编号
- 所属项目
- 版本号
- 环境
- 发布时间
- 发布人
- 发布结果
- 回滚标记
- 说明

## 6. 核心流程

### 6.1 需求到任务流程

```text
创建需求
  -> 评估优先级
  -> 排入阶段或迭代
  -> 拆解任务
  -> 执行开发
```

规则：

- 需求必须属于项目
- 需求可直接拆任务，也可先进入需求池待排期
- 需求关闭前应满足验收标准

### 6.2 测试流程

```text
创建测试计划
  -> 明确测试范围
  -> 执行测试
  -> 提交测试报告
  -> 结论反馈到需求 / 缺陷 / 发布
```

规则：

- 测试计划至少关联项目和测试范围
- 测试报告应汇总缺陷数量和测试结论
- 高风险测试结论应进入项目预警

### 6.3 发布流程

```text
登记版本
  -> 确认测试结论
  -> 发起发布
  -> 记录发布结果
  -> 必要时记录回滚
```

规则：

- 发布记录至少关联项目和版本
- 生产发布建议保留审批或确认记录
- 发布失败或回滚应自动进入项目日志和预警

## 7. 一期边界

一期做：

- 需求、迭代、测试计划、测试报告、发布记录基础对象
- 与项目、任务、缺陷、里程碑的关联
- 基础状态流转、通知和留痕

一期不做：

- 完整测试用例平台
- 自动化测试平台
- CI/CD 流水线编排
- 分支策略管理
- 多环境发布编排中心

## 8. 集成边界建议

研发工具链在一期只定义挂接边界：

- GitLab / Azure DevOps：需求、缺陷、提交、版本信息挂接
- Jenkins：构建结果、发布时间挂接
- 测试平台：测试结果摘要挂接

原则：

- 外部工具已有主数据时，本系统以引用和汇总为主
- 本系统保留项目管理口径下的结构化摘要和留痕

## 9. 本文件结论

一期的研发交付能力重点不是把研发工具全部搬进来，而是把需求、测试、发布真正纳入项目主线，让 PM、PMO 和研发团队能看见同一条交付链路。
