# 核心业务流程设计

## 1. 文档目的

本文件用于定义一期必须跑通的核心业务流程，作为原型、接口和状态机设计的基础。

## 2. 核心流程总览

### 2.1 流程设计约束

- 本文件只展开一期详细流程，不再重复说明“一期做什么范围”，分期范围统一以顶层 `04` 为准
- 预算、合同、验收、开票、回款在一期统一按项目侧跟踪流程设计，不做财务核算
- 评论、附件、日志、通知视为流程闭环的一部分，不视为可有可无的配套能力
- Web 是一期唯一业务主端，桌面端如果存在，也只承担交付壳职责
- 流程设计以最小可执行为原则，先保证闭环，再考虑增强分析能力
- 本文件中的主流程主要属于项目执行区，预算和合同相关流程在一期以经营管理区的轻量挂接方式存在
- 需求、测试发布、会议治理和资料归档在一期也进入主线，但详细规则分别由 `05`、`06`、`07` 承接

一期至少需要打通以下 12 条流程：

1. 项目立项流程
2. 项目规划流程
3. 任务执行流程
4. 需求与迭代流程
5. 测试与发布流程
6. 风险流程
7. 问题流程
8. 变更流程
9. 预算流程
10. 合同执行与回款流程
11. 会议纪要与行动项流程
12. 资料归档与审计流程

## 3. 项目立项流程

### 3.1 目标

把项目从“口头启动”变成“有编号、有负责人、有审批记录”的正式项目。

### 3.2 参与角色

- 发起人
- 项目经理
- PMO
- 审批负责人

### 3.3 流程步骤

```text
创建项目草稿
  -> 填写立项信息
  -> 提交立项审批
  -> 审批通过
  -> 自动生成项目编号
  -> 项目进入进行中 / 待规划
```

### 3.4 一期规则

- 草稿状态允许反复编辑
- 提交审批后锁定关键字段，变更需走修改流程
- 审批通过后自动生成项目编号
- 审批驳回后退回草稿并保留驳回意见
- 立项通过后默认创建项目空间和基础权限

## 4. 项目规划流程

### 4.1 目标

让项目从“只有名称”变成“有阶段、有里程碑、有负责人分工”的可执行计划。

### 4.2 流程步骤

```text
项目创建完成
  -> 选择项目模板或手工建计划
  -> 配置阶段
  -> 配置里程碑
  -> 拆解任务
  -> 指定负责人与计划时间
  -> 项目正式进入执行
```

### 4.3 一期规则

- 项目模板允许带出阶段、里程碑、标准任务
- 至少存在一个阶段和一个里程碑才能进入稳定执行状态
- 任务必须归属项目，可选归属阶段或里程碑
- 里程碑必须设置计划日期和责任人

## 5. 任务执行流程

### 5.1 目标

让 PM 和成员在系统内完成任务分配、推进、阻塞、完成和关闭。

### 5.2 流程步骤

```text
创建任务
  -> 指派负责人
  -> 开始执行
  -> 过程中可补充评论、附件、工时、协作人
  -> 若阻塞则标记阻塞并记录原因
  -> 完成后提交结果
  -> 关闭
```

### 5.3 一期规则

- 任务负责人可以更新执行状态
- PM 可以改派负责人和调整时间
- 阻塞状态必须填写阻塞原因和解除计划
- 已完成任务允许关闭前补充验收说明
- 已关闭任务默认只允许管理员或 PM 重新打开

## 5A. 需求与迭代流程

### 5A.1 目标

让需求、迭代和任务形成结构化链路，避免研发执行脱离项目主计划。

### 5A.2 流程步骤

```text
创建需求
  -> 评估优先级
  -> 排入阶段或迭代
  -> 拆解任务
  -> 跟踪开发与测试状态
  -> 关闭
```

### 5A.3 一期规则

- 需求必须归属项目
- 需求可挂阶段，也可挂迭代
- 需求关闭前应有验收标准或交付说明
- 迭代的详细规则以 `05-研发过程与质量交付设计` 为准

## 5B. 测试与发布流程

### 5B.1 目标

让测试和发布不只留在研发工具里，而能回到项目管理主线中留痕。

### 5B.2 流程步骤

```text
创建测试计划
  -> 执行测试
  -> 形成测试结论
  -> 登记发布记录
  -> 确认发布结果
```

### 5B.3 一期规则

- 测试计划至少关联项目和测试范围
- 发布记录至少关联项目、版本和环境
- 发布失败或回滚应进入项目日志和预警
- 详细字段和边界以 `05-研发过程与质量交付设计` 为准

## 6. 风险流程

### 6.1 目标

在问题真正发生前，把风险显式记录并跟踪应对动作。

### 6.2 流程步骤

```text
识别风险
  -> 评估概率与影响
  -> 制定应对措施
  -> 跟踪状态
  -> 解除或关闭
```

### 6.3 一期规则

- 风险必须有等级、责任人、计划处理日期
- 高风险自动进入 PMO 汇总视图
- 超过计划处理日期未关闭的风险进入预警

## 7. 问题流程

### 7.1 目标

让已发生的问题可以被定位、跟踪、升级和关闭。

### 7.2 流程步骤

```text
登记问题
  -> 指定责任人
  -> 跟进处理
  -> 需要时升级
  -> 处理完成
  -> 发起确认
  -> 关闭
```

### 7.3 一期规则

- 问题必须绑定责任人和期望解决日期
- 问题支持升级标记
- 超期未解决的问题进入预警
- 关闭前应记录处理结果

## 8. 变更流程

### 8.1 目标

控制范围、时间、成本等重大调整，避免项目执行失控。

### 8.2 流程步骤

```text
发起变更
  -> 填写变更内容与影响分析
  -> 评审
  -> 审批
  -> 执行落地
  -> 关闭
```

### 8.3 一期规则

- 关键变更必须审批
- 变更应记录对范围、时间、成本、资源的影响
- 变更批准后允许回写项目计划的关键字段
- 被拒绝变更保留历史记录，不可物理删除

## 9. 预算流程

### 9.1 目标

一期只做预算跟踪，不做财务核算。

### 9.2 流程步骤

```text
录入预算总额
  -> 提交审批
  -> 生效
  -> 录入实际投入
  -> 计算偏差
  -> 必要时发起调整审批
```

### 9.3 一期规则

- 一期预算只到项目级
- 已执行金额可手工录入，也可由外部系统同步
- 偏差率超过阈值自动预警
- 预算调整必须保留调整原因

## 10. 合同执行与回款流程

### 10.1 目标

把项目经营节点纳入跟踪，不做财务记账。

### 10.2 流程步骤

```text
登记合同
  -> 维护履约节点
  -> 维护验收节点
  -> 维护开票节点
  -> 维护回款节点
  -> 节点完成确认
  -> 逾期预警
```

### 10.3 一期规则

- 一个项目可绑定多个合同
- 每个节点至少包含计划日期、计划金额、状态
- 节点接近到期或逾期时自动触发预警
- 一期只跟踪状态与金额，不做账务核销

## 10A. 会议纪要与行动项流程

### 10A.1 目标

让晨会、周会和专题会中的结论、行动项和依赖形成结构化治理记录。

### 10A.2 流程步骤

```text
创建会议纪要
  -> 记录进展 / 阻点 / 决策
  -> 提取行动项
  -> 指定责任人与日期
  -> 跟踪完成状态
```

### 10A.3 一期规则

- 会议纪要可挂接项目、阶段或迭代
- 行动项必须绑定责任人
- 阻点可转问题或风险
- 详细规则以 `06-会议纪要与治理闭环设计` 为准

## 10B. 资料归档与审计流程

### 10B.1 目标

让项目资料和合同资料不只作为附件存在，而能形成台账、版本和审计链路。

### 10B.2 流程步骤

```text
登记资料台账
  -> 上传资料版本
  -> 维护分类与密级
  -> 归档
  -> 预览 / 下载 / 导出留痕
```

### 10B.3 一期规则

- 合同和经营资料应支持单独授权
- 已归档资料默认只读
- 下载、导出必须留痕
- 详细规则以 `07-资料档案与合规审计设计` 为准

## 11. 项目结项流程

### 11.1 目标

让项目结束有正式动作、有材料沉淀、有历史留痕。

### 11.2 流程步骤

```text
确认项目主要工作完成
  -> 检查未关闭任务 / 风险 / 问题 / 变更
  -> 填写结项说明与复盘
  -> 提交结项审批
  -> 项目结项
```

### 11.3 一期规则

- 未关闭关键事项时允许提示但不一定强制阻断
- 结项审批通过后项目转为已结项
- 已结项项目默认只读，特殊权限可重新打开

## 12. 流程设计输出要求

基于本文件，下一步需要为每条流程补以下材料：

- 页面流转草图
- 状态流转图
- 关键字段必填规则
- 通知触发点
- 审批触发点
- 日志记录点
- 与 `05`、`06`、`07` 的对象和页面映射关系

## 13. 本文件结论

一期真正要交付的不是模块目录，而是这些流程的闭环。只有流程跑通，项目管理软件才会被持续使用。
