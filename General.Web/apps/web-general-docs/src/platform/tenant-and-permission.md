# 租户与权限

General Admin 的权限设计分为三层：租户边界、角色权限、数据范围。

## 租户边界

Host 用户没有租户，负责平台级治理。普通业务用户属于具体租户，只能在租户上下文内访问业务数据。

这意味着 Host 用户在默认状态下不应该被当成某个租户成员。需要查看或维护租户内数据时，应进入明确的租户运维上下文。

Host 查看用户管理时有一个特殊分支：当当前租户为空并且当前用户是 Host 管理员，用户列表会禁用多租户过滤，从 `TenantId = null` 的 Host 上下文查询跨租户用户。这个能力只解决“Host 能看见系统账号”问题，不表示 Host 自动拥有每个租户内的业务数据权限。

## 角色权限

角色权限控制“能不能进入页面”和“能不能执行操作”。

- 页面查看权限用于进入页面、加载列表、查看详情。
- 管理权限用于新增、编辑、删除、启停、重置密码等写操作。
- 下拉选项和辅助查询应按页面需要授权，避免因为缺少管理权限导致只读页面报错。

平台菜单里有一类典型拆分：

- `Platform.Tenant.View`：进入租户页面和查看租户列表。
- `Platform.Tenant.Manage`：新增、编辑、删除租户，以及进入租户运维。
- `Platform.User.Manage`：新增、编辑、删除用户、重置密码和在线用户强制下线。
- `Platform.Scheduler.View`：查看定时任务列表、看板、节点和执行记录。
- `Platform.Scheduler.Manage`：新增、编辑、启停、删除任务和触发器。
- `Platform.Scheduler.Execute`：手动执行任务。

## 数据范围

数据范围控制“能看到哪些数据”。

- 全部数据：适合租户管理员、PMO 等治理角色。
- 本组织及下级：适合部门负责人和项目经理。
- 仅本人：适合普通成员。
- 自定义范围：适合跨部门协作或临时授权。

组织架构和用户管理都依赖数据范围。普通用户默认只能看到自己所在组织及下级组织，成员类角色默认更收敛，只能看到自己或与自己直接相关的数据。管理员和 PMO 这类治理角色才适合配置为全部数据。

组织架构页面也是这个逻辑：不是拿全量组织树直接展示，而是先根据当前角色和组织归属计算可见组织集合，再返回树结构。管理组织的按钮和新增、移动、删除接口仍由 `Platform.Organization.Manage` 控制。

## 当前约定

- `admin`：租户内管理角色，默认可看全部数据。
- `pmo`：项目治理角色，默认可看全部项目和经营视图。
- `pm`：项目经理，默认按组织范围、项目负责人、发起人、项目成员关系可见。
- `member`：项目成员，默认看自己参与的项目。
- `viewer`：经营查看角色，按授权范围查看经营数据。
